# Nodewire — Scaffolding Design

**Date:** 2026-05-12
**Status:** Approved
**Scope:** Initial project scaffolding only. Logic engine, node editor GUI, linking system, VS-aware link resolver and Create integration are out of scope for this spec — they will get their own specs.

---

## Purpose

Set up a buildable, runnable Forge 1.20.1 Kotlin mod project named **Nodewire** so that subsequent feature work has a stable foundation. After this step:

- `./gradlew build` produces a valid mod jar.
- `./gradlew runClient` launches Minecraft with the user's existing modset (via symlinked `run/mods`).
- A placeholder `logic_block` is registered and obtainable in creative inventory, proving the registration pipeline works.

This spec deliberately does NOT design the mod's gameplay features. It designs the *project* the gameplay features will live in.

## Compile-time dependencies

| Coordinate (approx, resolved at scaffolding time) | Scope | Notes |
|---|---|---|
| `thedarkcolour:kotlinforforge:4.12.0` | `implementation` | hard-dep, declared in mods.toml |
| `org.valkyrienskies:valkyrienskies-120-forge:2.4.10+a7a0898ae1` | `implementation fg.deobf(...)` | hard-dep, mod core depends on VS API |
| `com.simibubi.create:create-1.20.1:0.5.1.j-all` | `implementation fg.deobf(...)` | hard-dep for Create integration; exact version aligned to instance's `create-1.20.1-6.0.8.jar` if available, otherwise nearest published artifact (Create's maven publishing has historically lagged — fall back to `cursemaven` or `modmaven` coordinate if `tterrag` lacks 6.0.8) |
| `mezz.jei:jei-1.20.1-forge:15.20.0.x` | `compileOnly` | tooltips/integration only, not required at runtime |
| `curse.maven:jade-...` | `compileOnly` | tooltips only |

**Drivebywire is NOT added as a compile dependency.** Nodewire builds its own linking system; drivebywire is referenced only conceptually. It still loads at runtime via the symlinked instance mods.

**Version resolution fallback:** if a precise version is unpublished on the primary maven (common for Create and VS pre-release builds), use `cursemaven` with the project + file ID. The user's instance has the canonical jars — file IDs can be retrieved from CurseForge or Modrinth.

## Runtime / dev environment

Run configurations defined in `build.gradle.kts`:

```kotlin
minecraft {
    mappings("official", "1.20.1")
    runs {
        create("client") {
            workingDirectory(project.file("run"))
            property("forge.logging.markers", "REGISTRIES")
            property("forge.logging.console.level", "debug")
            mods { create("nodewire") { source(sourceSets.main.get()) } }
        }
        create("server") { /* same shape */ }
    }
}
```

After first `./gradlew genIntellijRuns && ./gradlew runClient`, the `run/` directory is created. A symlink replaces `run/mods`:

```bash
ln -s "/home/nitka/.local/share/PrismLauncher/instances/Total Valkyrien/minecraft/mods" \
      "/home/nitka/CODING/nodewire/run/mods"
```

**Consequence:** the dev client loads every jar in the user's PrismLauncher instance. Nodewire itself is loaded from source via `mods { create("nodewire") { source(sourceSets.main.get()) } }`, so it must NOT also exist as a built jar inside the symlinked dir — otherwise we get a duplicate-mod load failure.

**Sandbox isolation note:** the symlinked `run/mods` is a real, shared directory. Any worlds, configs, or per-mod data written by the dev client go into `run/` (saves, config/, logs/) which is *not* shared with the instance. Only the mods themselves are shared. This is intended.

## Entry point (Nodewire.kt) — minimal shape

```kotlin
@Mod(Nodewire.ID)
object Nodewire {
    const val ID = "nodewire"
    private val LOG = LogUtils.getLogger()

    init {
        val bus = FMLJavaModLoadingContext.get().modEventBus
        Registry.register(bus)
        LOG.info("Nodewire loading")
    }
}
```

`Registry.kt` declares one `DeferredRegister<Block>` and one `DeferredRegister<Item>`, registering a single `logic_block` (a vanilla `Block` with `Properties.copy(Blocks.IRON_BLOCK)` — placeholder, no BlockEntity yet).

## Acceptance criteria

The scaffolding step is complete when ALL of the following hold:

1. `~/CODING/nodewire/` exists as a git repository with an initial commit.
2. `./gradlew build` exits 0 and produces `build/libs/nodewire-0.1.0.jar`.
3. `./gradlew runClient` launches Minecraft 1.20.1 with Forge 47.4.10. The client reaches the main menu without crashes.
4. The user's existing modset (from the Total Valkyrien instance) is loaded — confirmed by mod count in the Mods screen matching the instance jar count.
5. `logic_block` appears in the creative tab (any tab is fine for now) and can be placed in a creative world.
6. The block's English name reads "Logic Block" (from `en_us.json`).
7. No duplicate-mod errors in `logs/latest.log`.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Create artifact version 6.0.8 not on tterrag maven | fall back to cursemaven coordinate `curse.maven:create-328085:<fileId>`; user's instance has the jar to extract fileId from |
| VS maven coordinate naming differs across versions (`valkyrienskies-120-forge` vs `valkyrienskies-forge-1201`) | resolve once during implementation, hard-code; if both fail, use cursemaven |
| Symlinked `run/mods` contains a built copy of Nodewire (from a prior install) | implementation step manually greps the target dir for `nodewire*.jar` before creating the symlink and aborts if found |
| Forge 47.4.10 MDK uses Gradle wrapper version mismatched with our 8.8 | pin wrapper distribution URL explicitly in `gradle-wrapper.properties` |
| ForgeGradle 6 incompatibility with Kotlin 2.0 | KFF 4.12 officially supports Kotlin 2.0; if compile fails, fall back to Kotlin 1.9.24 |
